# Vehicle Backend Changes

This note documents the latest backend changes for the vehicle management flow in `D:\SWP391_FE\BE`.

## Scope

Changed files:

- `API/Controllers/VehicleController.cs`
- `API/DTOs/VehicleDto.cs`

No database schema was changed in this update.

## Why this change was needed

The previous flow still depended on `VehicleCode` being typed by the user when creating a vehicle. That is not a good business fit because:

- `vehicle_id` is already generated by the database.
- `vehicle_code` is an internal system identifier.
- users should not need to know or maintain the numbering convention.

The backend now generates `VehicleCode` automatically while keeping the existing database structure unchanged.

## Current API behavior

### `GET /api/Vehicle`

Purpose:

- return the vehicle list for manager, admin, and coordinator screens

Behavior:

- accepts optional `status`
- valid values are only `AVAILABLE`, `INUSE`, `MAINTENANCE`
- returns `latitude` and `longitude` in the list response so the frontend can reuse the same payload for map-based management

### `GET /api/Vehicle/{id}`

Purpose:

- return one vehicle for the edit popup

Behavior:

- includes the same fields used by the management form, including `currentLocation`, `latitude`, `longitude`, and `lastMaintenance`

### `POST /api/Vehicle`

Purpose:

- create a new vehicle

Business rules:

- `VehicleCode` is generated inside the backend
- duplicate `LicensePlate` is rejected before insert
- `VehicleTypeId` must exist
- only `AVAILABLE` or `MAINTENANCE` can be set manually on create
- `CurrentLocation`, `VehicleName`, and `LicensePlate` are trimmed before save

Vehicle code generation:

1. Load the selected vehicle type.
2. Read existing vehicle codes of the same type.
3. Reuse the current prefix of that type if one already exists.
4. If that type has no vehicles yet, derive the prefix from `VehicleType.TypeCode`.
5. Normalize long type codes:
   - `HELICOPTER -> HELI`
   - `AMPHIBIOUS -> AMPH`
6. Find the highest numeric suffix and create the next code in the format `PREFIX-001`.

Examples:

- `BOAT-003 -> BOAT-004`
- `HELI-001 -> HELI-002`
- `AMPH-002 -> AMPH-003`

### `PUT /api/Vehicle/{id}`

Purpose:

- update basic vehicle information

Editable fields:

- `VehicleName`
- `VehicleTypeId`
- `LicensePlate`
- `Capacity`
- `Status`
- `CurrentLocation`
- `Latitude`
- `Longitude`
- `LastMaintenance`

Business rules:

- `VehicleCode` is not editable
- duplicate `LicensePlate` is rejected before save
- manual status changes are limited to `AVAILABLE` and `MAINTENANCE`
- a vehicle already in `INUSE` cannot be manually moved to another status
- `UpdatedAt` is refreshed on every successful update

### `DELETE /api/Vehicle/{id}`

Purpose:

- delete a vehicle

Business rules:

- vehicle deletion is blocked when status is `INUSE`
- related rows in `RescueOperationVehicles` are removed first to avoid foreign key failures

## DTO changes

### `CreateVehicleDto`

Removed:

- `VehicleCode`

Reason:

- it is now generated by the backend

### `UpdateVehicleDto`

Keeps:

- editable business fields only

Does not include:

- `VehicleCode`

Reason:

- `VehicleCode` remains an internal identifier and should stay stable after creation

## Validation summary

- invalid status filter on `GET /api/Vehicle?status=...` returns `400`
- duplicate license plate on create/update returns `400`
- unknown vehicle type returns `400`
- manual status outside `AVAILABLE` and `MAINTENANCE` returns `400`
- deleting `INUSE` vehicle returns `400`

## Frontend impact

The frontend no longer needs to ask the user for `VehicleCode` during create or edit. It should submit only business fields, while the backend handles the internal code generation and persistence.
